Allgemeines

Die axs_xlink Klasse ermöglicht die Ansteuerung  von AX-Felbuscontrollern und von Lenzing SlaveNote Modulen über eine Ethernetkarte. Über die verwendete Ethernetkarte können keine normalen TCP/IP Verbindungen mehr geöffnet werden.  Die Xlink-Module können zu einer Kette zusammen geschaltet werden oder über einen Switch als Netzwerk betreiben werden. angeschlossen werden. Weiters werden auf redundanten AX-Sever-Paaren über diese Schnittstelle Redundanzpakete versand.

Graphical Representation

Als Instanzname muss "Xlink_?" für die Klasse verwendet werden. Das Fragezeichen steht für eine Zahl von 1 bis 9. Falls nur ein XlinkMaster verwendet wird sollte beim Instanznamen "Xlink_1" begonnen werden.

Installation der Netzwerkstreiber unter Windows

Der Netzwerkstreiber heißt axnpf.sys und befindet sich im ./bin Verzeichnis von automationX. Er wird beim ersten Start des Xlink-Masters automatisch installiert. Dabei wird der Windows-Registry-Eintrag HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\AXNPF angelegt. Durch löschen dieses Eintrages und einem Neustart des Rechners kann der Treiber reinstalliert werden.

Damit das Echtzeitverhalten der Xlink-Verbindung nicht gestört wird müssen die IP-Dinste auf der Ethernet-Karte abgeschalten werden. Dazu führen sie folgende Schritte aus:

  • Öffnen sie den Explorer und wählen sie Systemsteuerung/Netzwerkverbindungen aus.
  • Machen sie einen Doppelklick auf die LAN-Verbindung die für Xlink vorgesehen ist.
  • Drücken sei Eigenschaften.
  • Wählen sie alle Elemente für die Verbindung ab und drücken sie Schließen:

 

Installation der Netzwerkstreiber unter Linux

Unter Linux ist kein Netzwerktreiber notwendig. Es ist möglich direkt auf eth0 bis eth7 zu zu greifen. Damit das Echtzeitverhalten der Xlink-Verbindung nicht gestört wird müssen die IP-Dinste auf der Ethernet-Karte abgeschalten werden. Dazu führen sie folgende Schritte aus:

  • Öffnen sie die Datei /etc/sysconfig/network-scrips/ifcfg-eth?
  • Öffnen löschen sie bei folgenden Einträgen die IP-Adddressen:
             IPADDR=
         NETMASK=
         NETWORK=    
             GATEWAY=
         BROADCAST=
  • Setzen sie den Parameter BOOTPROTO auf none
  • Zuletzt müssen die Etherner-Schnittstellen neu gestartet werden:
            /etc/init.d/network restart
     

Installation der Klasse

Für die Installation der Klasse öffnen sie das Projekt, und wählen in der Konfiguration den Tree axsystem aus. Hier fügen sie die Klasse axs_xlink aus dem Pfad .\classlib\systemax\axserver\axs_interface\axs_xlink ein. Für die Klasse kann ein beliebiger Instanzname vergeben werden.


Alls erstes muss unter EthCardA der Name benutzten Ethernetkarte eingetragen werden (z.B. eth1) oder die MAC-Adresse einer Karte. Für redundante Systeme muss hier eth0 oder eth1 oder ... eingetragen werden. Unter Windows kann die MAC-Adresse über die Datei ./cache/rdip/local_ethernet_<host>.mac einer Ethernetkarte (eth??) zugeordnet werden. (<host> steht für den Hostnamen des Rechners) Diese Datei hat folgende Form:

eth0 = 00-50-BA-E5-E4-B9
eth1 = 00-30-4A-67-9A-3F

Mit dem Tool EthInfo.exe können die Etherneteinstellungen bearbeitet werden. Klicken sie einfach mit der rechten Maustaste auf das Icon im linken oberen Eck. Und wählen sie "MAC-Adressen konfigurieren".

Wenn Linux verwendet wird ist diese Datei nicht notwendig. Der Treiber holt sich die MAC-Adressen direkt von der Systemkonfiguration. Die MAC-Adressen können unter Windows mit  ipconfig /all abgefragt werden. (siehe bei "Physikalische Adresse"), unter Linux kann ifconfig verwendet werden.

Wollen sie einen redundanten Ring mit Xlink aufbauen, so ist unter EthCardB die Ethernetkarte einzutragen über die der Ring geschlossen wird.

Unter Windows können sie mit dem Tool EthInfo.exe nähere Informationen über die eingebauten Netzwerkskarten abrufen:


 

Bei allen aXlink100 müssen sie über die Enumeratorvariabel  XlinkMaster auswählen an welcher Xlink-Verbindung der aXlink100 angeschlossen ist.
 

Funktion der Klasse

Die Klasse enthält folgende Element:

Variabel

Funktion

EthCardA

Ist der Name der benutzen Ethernetkarte eth? oder die MAC-Adresse der Ethernetkarte.

EthCardB

Bei einem redundanten Ring ist es der Name der zweiten Ethernetkarte eth? oder die MAC-Adresse, bzw. ist diese Variabel leer zu lassen wenn kein Ring vorhanden ist.

Reset

Mit diesem Flag kann der Xlink-Master resetiert werden.

EthernetCardValidA

Dieses Flag wird auf TRUE gesetzt wenn die angegebene erste Ethernetkarte geöffnet werden konnte.

EthernetCardValidB

Dieses Flag wird auf TRUE gesetzt wenn die angegebene zweite Ethernetkarte geöffnet werden konnte.

LinkErrorA

Dieses Flag wird auf TRUE gesetzt wenn kein Ethernet-Kabel an der ersten Ethernetkarte angeschlossen ist.

LinkErrorB

Dieses Flag wird auf TRUE gesetzt wenn kein Ethernet-Kabel an der zweiten Ethernetkarte angeschlossen ist.

State

Ist der aktuelle Zustand des Masters. Eine Beschreibung zu den einzelnen Zuständen findet man weiter unten.

HasRing

Dieses Flag geht auf 1 wenn die Ethernet-Karte-B für einen Ring benutzt wird.

RingOk

Dieses Flag wird auf TRUE gesetzt wenn der Master erkennt das Telegramme ordnungsgemäß im Kreis laufen.

RdLifetick

Dieses Flag wird auf TRUE gesetzt wenn vom anderem AX-Server ein Redundanz-Lifetick auf dieser Xlink-Verbindung empfangen wird.

Debug

Über diese Bits kann eingestellt werden welche Debug-Meldungen der Maser in den Runtime-Messages ausgeben soll.
Bit 0 (0x01) = Bus-Scans anzeigen

DevicesOnBus

Ist die Anzahl der gefunden über Geräte am Bus. Geräte mit doppelten Adressen werden nur einmal gezählt.

Devices

Ist ein Array das Informationen zu allen Geräten die gefunden wurden liefert. Der Offset im Array bestimmt die Geräte Adresse. Ist ein Eintrag Null, so wurde dieses Gerät am Bus nicht gefunden.
Bit 0 = Das Gerät hat eine Verbindung auf Link A
Bit 1 = Das Gerät hat eine Verbindung auf Link B
Bit 2 = Nur für interne Zwecke
Bit 3 = Das Gerät ist aktiviert.
Bit 4 = unbenutzt
Bit 5 = unbenutzt
Bit 6 = unbenutzt
Bit 7 = Ein weiteres Gerät mit der gleichen Adresse ist am Bus vorhanden.

ErrorString

Enthält einen Text mit der aktuellen Fehlermeldung.

MacAddressUsedA

Hier wird die MAC-Adresse der benutzten ersten Ethernetkarte angezeigt.

MacAddressUsedB

Hier wird die MAC-Adresse der benutzten zweiten Ethernetkarte angezeigt.

Der aktuelle Zustand des Masters kann auf der Klassenansicht im Feld "State" abgelesen werden. Wenn die LED im linken oberen Eck rot ist dann konnte die Ethernetkarte nicht geöffnet werden.

Der Xlink-Master kennt folgende Zustände:

OK: Das ist der Betriebszustand in dem alles in Ordnung ist.

Link_Error: Dieser Zustand passiert wenn bei einer Ethernetkarte das Kabel nicht angeschlossen ist.

NO_Link: Dieser Zustand passiert wenn bei keiner Ethernetkarte ein Kabel angeschlossen ist.

Wrong_Address: Es liegt ein Fehler in der Konfiguration der angeschlossenen Felbuscontroller bzw. SlaveNotes vor. Bei einem dieser Geräte ist die Adresse auf 0,254 oder 255 gestellt. In den Runtime-Memessages wird die MAC-Adresse des betroffenen Gerätes angezeigt. Mit dem Tool EthScanner.exe können sie die Netzwerkstopologie ansehen.

Double_Address: Es liegt ein Fehler in der Konfiguration der angeschlossenen Felbuscontroller bzw. SlaveNotes vor. Bei mindestens zwei dieser Geräte ist eine Gleiche eingestellt. In den Runtime-Messages wird die Adresse und die MAC-Adresse des betroffenen Gerätes angezeigt. Mit dem Tool EthScanner.exe können sie die Netzwerkstopologie ansehen.

Ethernet_Error: Dieser Zustand passiert wenn die Ethernetkarte nicht geöffnet werden konnte. Prüfen sie in diesem Fall MacAddress oder die Datei ./cache/rdip/local_ethernet_<host>.mac. Unter Linux müssen die MAC-Addresen mit den Werten von ifconfig übereinstimmen.

Remote_Error: Dieser Zustand passiert wenn die MAC Adressen des anderen AX-Servers in einem redundanden System nicht konfiguriert sind. Prüfen sie in diesem Fall die Datei ./cache/rdip/remote_ethernet_<remote-host>.mac. Diese Datei hat folgende Form:

eth0 = 00-50-BA-E5-E4-B9
eth1 = 00-30-4A-67-9A-3F

 

Die beiden obersten LED zeigen an welche Ethernetschnittstellen von der Klasse benutzt werden. Die Farbe bestimmt den Zustand der Ethernetschnittstelle:

rot = Ethernetkarte konnte nicht geöffnet werden
gelb = Das Kabel an der Ethernetkarte ist ab gesteckt
grün = Alles ok

Die LED Devices zeigt an wie viele Geräte am Bus gefunden wurden. Sie wechselt auf gelb wenn keine Geräte gefunden werden konnten.

Die Ring-LED zeigt an ob der Ethernetring geschlossen ist. Die Prüfung ob der Ring geschlossen ist, erfolg nur alle 30 Sekunden.

Wenn der Xlink-Master nicht in den OK-Zustand kommt, ist es nützlich zu wissen was das Problem verursacht. Dazu sollte man die Debug Variabel auf einen Wert 1 setzen und das Runtime-Message-Fenster öffnen. Danach wählen sie "Xlink" aus:

Eine weitere Möglichkeit ist das Tool EthScanner.exe. Mit ihm kann die Netzwerkstopologie angezeigt werden:

 

Sollten Probleme bei der Redundanzverbindung vorhanden sein, prüfen sie die Dateien ./cache/rdip/remote_ethernet_<remote-host>.mac auf beiden Rechnern. Am Master müssen die MAC-Adressen der Ethernetkarten des Standby-Rechners eingetragen sein, und umgekehrt. Mit dem Tool EthInfo.exe können sie die MAC-Adressen des eigenen Rechners auslesen:

Mit EthInfo.exe können auch die Remote-MAC-Adressen konfiguriert werden. Klicken sie einfach mit der rechten Maustaste auf das Icon im linken oberen Eck. Und wählen sie "Remote Adressen konfigurieren".

Ein weiteres Programm ist  EthXlinkScanner.exe. Damit kann der Xlink-Datenverkehr auf einer Neztwerkskarte analysiert werden.

 

Firmwareupgrade

Mithilfe des Tools Ethscanner kann man ein Firmwareupgrade auf den aXlink100 machen.

Man öffnet dazu den EthScanner:

Klickt man nun doppelt auf einen der angezeigten aXlinks100 öffnet sich folgendes Popup:

In der Anzeige wird die aktuelle Firmware mit dem Datum angezeigt. Weiters werden die Macadressen angezeigt.

Das Firmwarefile auf dem Xlink heißt netx.rom und kann nicht umbenannt werden.

Mit dem Button Choose wählt man jetzt die Firmware aus und mit dem Button Upload kann man die neue Firmware dann einspielen.

Nach dem Upload ist noch ein Reset nötig, damit die neue Firmware geladen wird. Es kann auch sein das ein zusätzliches Activate notwendig ist.

Nur in einem Subnetz

Die Topologie Netzwerk und Kette kann auch kombiniert werden, siehe heir. Im Xlink-Netzwerk  können handelsübliche Switches verwendet werden, sofern das Xlink-Netz nicht mit einem IP-Netz kombiniert wird.  Beim Anschluss der Xlink-Greäte ist es egal ob X1 oder X2 benutzt wird. Ebenso ist es egal ob man eins-zu-eins Netzwerkskabel oder ausgekreuzte Kabel verwendet.

Die folgenden Bilder zeigen wie einzelne Geräte zusammen geschalten werden können:

 

Kette

Bei einer Kette sind alle XLink-Geräte in einer Daisy-Chain-Line zusammen geschalten, und nur über eine Netzwerkskarte an einem AX-Rechner angeschlossen.

Ring

Bei einem Ring sind alle XLink-Geräte in einer Daisy-Chain-Line zusammen geschalten. Das Ende der Linie wird auf die zweite Netzwerkskarte am AX-Rechner zurückgeführt. Bei einem redundanten Aufbau mit Master und Standby, müssen die beiden AX-Rechner über Switches mit der Daisy-Chain verbunden werden.

 

 

Netzwerk

Bei einem Netzwerk ist jedes Xlink-Gerät an einem Port eines Switches angeschlossen. Damit lassen sich alle Arten von Baumtopologien realisieren.

Netzwerk (redundant)

Bei einem rdundanden Netzwerk ist jedes Xlink-Gerät an einem Port eines Switches angeschlossen (grün). Der zweite Netzwerksanschluss der Xlink-Geräte wir an ein separates Subnetz angeschlossen (blau). Wenn nun ein Subnetz ausfällt, kann das zweite Subnetz die Kommunikation immer noch aufrecht erhalten.

Netzwerk mit Unterketten

Bei einer Netzwerktopologie können auch einzelnen Ketten aus Xlink-Geräten an die Port der Switches angehängt werden.

 

Xlink-Netze mit Firmennetzen verbinden

Wenn ein Xlink-Netzwerk mit einem normalen Firmen-Ip-Netzwerk gemischt werden soll, so müssen zwei VLAN's erstellt werden. Ein VLAN für den Xlink-Datenverkehr und das andere VLAN für den IP-Datenverkehr. Die eingesetzten Switches müssen daher VLAN fähig sein, und müssen so konfiguriert werden das der Xlink-Datenverkehr die höchste Priorität hat. Sollten die Switches diese beiden Optionen nicht unterstützen so kann für die ordnungsgemäße Funktion des Netzwerkes nicht garantiert werden. (näheres siehe in der aXlink100 Dokumentation)

Auf den AX-Servern muss für die Xlink-Verbindung immer eine extra Netzwerkskarte verwendet werden.
