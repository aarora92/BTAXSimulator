Variable Speed Motor

Features | Installation | User Interface | Configuration | Implementation Notes
 
 

Features

This class can be used for all variable speed motors. It features the following:

  • Fully Integrated Control (based on IEC 1131-3 Function Block Diagram)
  • Field Links: On/Off (Binary Outputs, Mandatory), Speed Set (Binary output, Manditory), Run Feedback (Binary Input, Optional), Ready Feedback (Binary Input, Optional), Remote Switch (Binary Input, Option), Speed (Binary Input, Optional).
  • Integrated Simulation Mode + Feedback Emulation
  • Integrated Service and Maintenance Functions
  • Alarm Suppression for Invalid Field Links
  • Automated Generation Visualization of Interlock Conditions
  • Operator, Parameterization, and Service Pop-Up Windows
  • Trending
  • User Access Configuration for Individual Instances

 
 

Integrated Control

All control is fully integrated; no external PLC is required.

Field Links

The Variable Speed Motor requires two outputs: one On/Off switch and one Speed Set output value (the "on" and "set" variables). A binary input that indicates when the motor is actually running (run_in), that the motor is ready to be turned on (ready_in), that the motor is allowed to run (rem_in) and the current speed of the motor (speed_in). These variables are all found under the Server Data tab in the class configuration window.
 
 


 
 
 
 

Simulation Mode

In simulation mode the run feedback follows the On/Off signal with a variable delay time. In this mode the motor is disconnected from the field even if the On/Off Output is connected to an I/O Variable. An alternate mode (Feedback Emulation) does process the On/Off Output, but emulates individual inputs - thus ignoring their actual states.

Service and Maintenance Functions

To allow maintenance personel an insight into failure frequency, number of activations and operational time, the Motor Class integrates service and maintenance functions together in the Maintenance pop-up window. To notify the operator, a service alarm is generated automatically and an icon is displayed beside the motor's graphical representation and on the Operator pop-up window..
 
 


 
 
 
 

Alarm Suppression for Invalid Field Links

The Motor Class checks validity of connected I/O. If a field link is interrupted but the I/O status is normal, the motor class will suppress any alarms.

System-Generated List of Interlock Conditions

An interlock indication appears in the Operator Pop-up window when an interlock condition is applied to a motor. To track the condition, the operator can simply select the interlock indication. The list is generated automatically and presented in graphical form; no configuration is necessary for this feature.

Pop-Up Windows

The Motor Class contains pop-up windows for Operation, Maintenance and Parameterization. These windows are already built into the class and no further configuration is necessary to access them. The Operator Interface pop-up window is accessed by selecting the Motor object itself. The Maintenance and Parameterization pop-up windows are accessed from the Operator Interface pop-up.

Trending

The Motor Class trends the output, on/off switch (on). The Trend function also works in simulation mode.

User Access Configuration for Individual Instances

User access for the following functions can be uniquely configured for each instance of the Motor class:

  • Trending
  • Operations
  • Interlock
  • Service & Maintenance
  • Parameterization
See Configuration for details.

To install the Motor class, all of the above files must be copied into a common directory within the project's "classlib" directory. The name of the directory can be chosen freely. For example (refer to the directory structure below), to add the Motor class to an existing class library simply follow the directory structure in the automationX program directory: AutomationX\projects\my_project\classlib\ProcessIndustry\motor\motor_var_speed

Parameterization and Mode Pop-Up

  • The Parameter section allows setup of the motor's maximum response time for Run feedback, Speed signal feedback and Current signal feedback. The default value for each is five seconds.
  • The Modes section allows switching to (local) Simulation Mode. In this mode all inputs are simulated and the output is disconnected from the field. The Overwrite Interlock Mode connects the auto_on variable with the field regardless of the status of the interlock variable. If niether of these two Check Boxes is activated the motor is in standard mode.
  • The Emulations section allows individual emulation of field inputs Run, Ready and Remote. The Run Input "follows" the output with a variable delay time (default = 3 seconds); the Ready and Remote feedbacks are considered ON.
 
Maintenance Pop-Up
 
  • The "Enable Maintenance" Button enables or disables all maintenance functions.
  • If one of the limits is exceeded the motor class launches a Service Request "Warning".
  • Each Limit can be reset individually, the reset date is automatically entered in the Last Reset field.

Configuration

I/O Configuration

The motor is linked to the field by filling out the reference fields of the following variables: on, set, run_in, rdy_in, rem_in and speed_in. Additional parameters are: the maximum time delay between an output command and the arrival of the proper feedback (error_time) and the limits for service request (see the Maintenance pop-up).

Notes:

  • If "run_in" is not connected, the class routes the output to the internal run variable. This fakes that the feedback is always OK. If "run_in" is connected, the field must provide the proper feedback.
  • If "rdy_in" is not connected, the class assumes that the motor is always ready.
  • If "rem_in" is not connected, the class assumes that the remote switch is always turned on.
  • The maximum delay time for the run feedback also can be changed in online mode in the Parameterization pop-up window.

 
 
 
Naming Conventions for Automated I/O Configurations

Normally, while implementing a class containing input and output variables (VAR_INPUT, VAR_OUTPUT), the Reference Fields (of these variables) need to be configured individually. However, if a class developer is required to use specific naming conventions, he or she may know the reference names in advance. For example, an organization’s internal standards may dictate tag creation according to the following conventions:

Tag name RUN FEEDBACK = Instance Name + "_" + "run"
 
 


 
 

automationX allows substitution addressing. Enter $Path_run in the appropriate reference field and the system automatically configures each instance according to the production rules of the class. For example, the instance name "VAR_MTR" would produce the reference field "VAR_MTR_run".
 

Linking the Motor to a Process

Instances of the motor class are linked into a process context by means of two variables:

  • auto_on is the condition that indicates the motor is supposed to run in Automatic Mode. The pre-conditions are that the motor actually is NOT in Manual Mode, and that no Interlock is applied on the motor.
  • lock is the condition that the motor MUST NOT run, either in Automatic nor Manual Mode. Use the Overwrite Interlock check box in the Parameter pop-up window to overwrite an interlock condition.
 

 
User Access Configuration for Individual Instances

It is possible to set up unique user permissions for individual instances of the Motor class. During configuration of an instance, locate the "access" variables "TrendAccess", "OperateAccess", "InterlockAccess", "ServiceAccess" and "ParameterAccess" in the Client Data list. Selection of these variables will open the user access settings in the attribute window. From here, choose the appropriate user permissions by enabling or disabling the check-box beside the user level.
 
 


 
 

An access variable is enabled when a user is logged in with that access level. Likewise an access variable is disabled if there is no user logged on with that access level. Access variables are attached to control elements (e.g. buttons) within the class, and are checked via ACL when deciding whether or not a user is allowed to use that control element. 

For example, the access_trend_docu variable in the previous figure shows Access Levels "Guest" and "Operator" as disabled. If a user is logged on using one of these levels and pressed the Trend button on the popup interface, access is denied according to the Button1Down callback code attached to the button.
 
 


 
 

Additionally, the access_trend_docu variable is attached to the SelectSet variable in the Client Data of the Trend button element - in this case a rectangle. When the TrendAccess variable is true, the button displays a full color trend pixmap indicating that the button is active and there is permission to use it. When the TrendAccess variable is false, the button displays a grey trend pixmap indicating that the button is inactive and permission to use it is currently denied.

Implementation Notes

Higher Class Sophistication through pre-evaluation of Inputs

The inputs RUN, REMOTE, and READY are connected to "run_in", "rem_in", and "rdy_in". However, the motor class works with internal variables "run", "rem", and "rdy". This approach allows manipulation of the variables if the simulation mode is on, or one or more input variables are not connected. The result is a much more sophisticated class.
 
 


 
 
 
 

Simulation

The Motor class has a simulation mode which can be enabled one of two ways:

  • Local Simulation (The class variable sim_local is ON, Activation through Parameter Pop-Up)
  • Global Simulation (The system variable "$$_simulation" variable is ON)

 
 

Suppression of Alarms in case of Invalid I/O

The Motor class avoids the generation of consecutive alarms in the event that some I/O loses proper connection to the field.
