Constant Speed Motor

Features | Installation | User Interface | Configuration | Implementation Notes
 


Features

This class can be used for all single speed motors. It features the following:

  • Fully Integrated Control (based on IEC 1131-3 Function Block Diagram)
  • Field Link: On/Off (Binary Output, Mandatory), Run Feedback (Binary Input, Option), Ready Feedback (Binary Input, Option), Remote Switch (Binary Input, Option).
  • Integrated Simulation Mode + Feedback Emulation
  • Integrated Service and Maintenance Functions
  • Depression of Alarm Generation in Case of Invalid Field Links
  • Automated Generation and Online Visualization of Interlock Conditions
  • Operator, Parameterization, and Service Pop-Up Windows
  • Trending


 
 

Integrated Control

All control is fully integrated; no external PLC is required.
 
 

Field Links

The Constant Speed Motor requires at least one binary output or On/Off switch (the "on" boolean variable). Additionally, it is equipped to process binary inputs that indicate when the motor is actually running (run_in), that the motor is ready to be turned on (ready_in) and that the motor is allowed to run (rem_in). These variables are all found under the Server Data tab in the class configuration window.
 
 


 
 
 
 

Simulation Mode

There is a simulation mode built in to the Motor Class such that the run feedback follows the On/Off signal with a variable delay time. In this mode the motor is not linked to the field even if the On/Off Output is connected to an I/O Variable! An alternate mode (Feedback Emulation) does process the On/Off Output, but emulates individual inputs - thus ignoring their actual status. Where the simulation mode is more applicable to early project development, the emulation mode allows to continue in a process, even if some individual inputs do not work properly.
 
 
 
 

Service and Maintenance Functions

In order to allow maintenance personal an insight into failure frequency, number of turn-ons and operational time, the Motor Class integrates certain service and maintenance functions together in the Maintenance pop-up window. To notify the operator, a service alarm is generated automatically and an according icon is then displayed beside the motor.
 
 


 
 
 
 

Depression of Consecutive Alarms in case of Invalid I/O

The Motor Class always checks the validity of the connected I/O. In a case where the field link is interrupted but the I/O status is normal, the motor class will depress the alarms.
 
 

System-Generated List of Interlock Conditions

An interlock indication will appear in the Operator Pop-up window when an interlock condition is applied to a motor. To track the condition, the operator can simply select the interlock indication. The list is generated (automatically) on-line and presented in graphical form; no configuration is necessary for this feature.
 
 

Pop-Up Menus

The Motor Class contains interface pop-up windows for Operation, Maintenance and Parameterization. These windows are already built into the class and no further configuration is necessary to access them. The Operator Interface pop-up window is accessed by selecting the Motor object itself. The Maintenance and Parameterization pop-up windows are accessed from the Operator Interface pop-up.
 
 

Trending

As a default setting, the Motor Class will trend the output (on/off) and the run feedback. The Trend function will work while in simulation mode as well.

Installation

To install the Motor class, all of the above files must be copied into a common directory within the project's "classlib" directory. The name of the directory can be chosen freely. For example (refer to the directory structure below), if we want to add the Motor class to an existing class library we would simply follow the directory structure in the AutomationX program directory: AutomationX\projects\my_project\classlib\ProcessIndustry\motor\motor_1speed

3. Parameterization and Mode Pop-Up

This pop-up menu is password protected!
 

  • The Parameter section allows an operator-friendly setup of the motor's maximum response time. The default value is five seconds.
  • The Modes section allows switching to (local) Simulation Mode. In this mode all inputs are simulated and the output is disconnected from the field! The Overwrite Interlock Mode connects the auto_on variable with the field no matter of the status of the interlock variable. This is useful when one MUST turn on the motor despite of interlock conditions. If none of the two Check Boxes is activated the motor is in standard mode.
  • The Emulations section allows an individual emulation of the field inputs in case they are INVALID. Then the Run Input "follows" the output with a variable delay time (default = 3 seconds). The Ready and Remote feedbacks are considered to be ON.

 

4. Maintenance Pop-Up
 

  • The "Enable Maintenance" Button enables or disables all maintenance functions.
  • If one of the limits is exceeded the motor class launches a Service Request "Warning".
  • Each Limit can be reset individually; then the date is entered in the according field.

Configuration

1. I/O Configuration

The Motor Class has only a few configuration parameters. First, the motor must be linked to the field. This is  done by filling out the reference fields of the following variables: on, run_in, rdy_in, rem_in. The other possible parameters are: the maximum time delay between a output command and the arrival of the proper feedback (error_time) and the limits for service request (compare maintenance pop-up).

Notes:

  • If "run_in" is not connected, the class routes the output to the internal run variable. This fakes that the feedback is always OK. If run_in is connected, the field must provide the proper feedback!
  • If "rdy_in" is not connected, the class assumes that the motor is always ready.
  • If "rem_in" is not connected, the class assumes that the remote switch is always turned on.
  • The maximum delay time for the run feedback also can be changed in online mode by means of the parameterization pop-up menu.

2. Naming Conventions for Automated I/O Configurations

Normally, while implementing a class containing input and output variables (VAR_INPUT, VAR_OUTPUT), the Reference Fields (of these variables) need to be configured individually. However, if a class developer is required to use specific naming conventions, he or she may know the reference names in advance. For example, an organization’s internal standards may dictate tag creation according to the following conventions:

Tag name RUN FEEDBACK = Instance Name + "_" + "RUN"

Tag name OUTPUT = Instance Name + "_" + "ON".
 
 


 
 

To accommodate this situation, AutomationX allows for a substitution addressing mechanisms. The class developer simply enters $Path_RUN and $Path_ON in the appropriate reference fields. The system will automatically configure each of the instances according to the production rules of the class. For example, the instance name "M1234" would produce the reference fields "M1234_RUN " and "M1234_ON".
 

3. Linking Motor into a Process Context

Instances of the motor class are linked into a process context by means of two variables:

  • auto_on is the condition that indicates that the motor is supposed to run in Automatic Mode. The pre-conditions are that the motor actually is NOT in Manual Mode, and that no Interlock is applied on the motor.
  • lock is the condition that the motor MUST NOT run, neither in Automatic nor Manual Mode. The only way to overwrite a Interlock Condition is a specific mode (compare to the parameter pop-up menu).

Implementation Notes

Higher Class Sophistication through pre-evaluation of Inputs

The inputs RUN, REMOTE, and READY are connected to "run_in", "rem_in", and "rdy_in". However, the motor class works with internal variables "run", "rem", and "rdy". This approach allows manipulation of the variables if the simulation mode is on, or one or more input variables are not connected. The result is a much more sophisticated class.
 
 


 
 
 
 

2. Simulation

The Motor class has a simulation mode which can be enabled by either one of two ways:

  • Local Simulation (The class variable sim_local is ON, Activation through Parameter Pop-Up)
  • Global Simulation (The system variable "$$_simulation" variable is ON) 


 
 

3. Depression of Alarms in case of Invalid I/O

The Motor class avoids the generation of consecutive alarms in the event that some I/O loses proper connection to the field.
