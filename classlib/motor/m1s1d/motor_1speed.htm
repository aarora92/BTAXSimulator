Single Speed Motor

Features | Installation | User Interface | Configuration | Implementation Notes
 
 

Features

This class can be used for all single speed motors. It features the following:

  • Fully Integrated Control (based on IEC 1131-3 Function Block Diagram)
  • Field Link: On/Off (Binary Output, Mandatory), Run Feedback (Binary Input, Option), Ready Feedback (Binary Input, Option), Remote Switch (Binary Input, Option).
  • Integrated Simulation Mode + Feedback Emulation
  • Integrated Service and Maintenance Functions
  • Alarm Supression for Invalid Field Links
  • Automated Generation and Online Visualization of Interlock Conditions
  • Operator, Parameterization, and Service Pop-Up Windows
  • Trending
  • User Access Configuration for Individual Instances
  • Visual Indication of Forced I/O



 
 

Integrated Control

All control is fully integrated; no external PLC is required.

Field Links

The Single Speed Motor requires at least one binary output or On/Off switch (the "on" boolean variable). Additionally, it is equipped to process binary inputs that indicate when the motor is actually running (run_in), that the motor is ready to be turned on (ready_in) and that the motor is allowed to run (rem_in). These variables are all found under the Server Data tab in the class configuration window.
 
 


 
 

Simulation Mode

There is a simulation mode built in to the Motor Class; the run feedback follows the On/Off signal with a variable delay time. In this mode the motor is not linked to the field even if the On/Off Output is connected to an I/O Variable. An alternate mode (Feedback Emulation) does process the On/Off Output, but emulates individual inputs - thus ignoring their actual status. Where the simulation mode is more applicable to early project development, the emulation mode allows continued process control even if some inputs do not work properly.

Service and Maintenance Functions

In order to allow maintenance personel an insight into failure frequency, number of activations and operational time, the Motor Class integrates service and maintenance functions together in the Maintenance pop-up window. To notify the operator, a service alarm is generated automatically and an icon is displayed beside the motor's graphical representation and the operator interface pop-up window.
 
 


 
 

Alarm Supression for Invalid Field Links

The Motor Class always checks the validity of connected I/O. In a case where the field link is interrupted but the I/O status is normal, the motor class will supress the alarm.

System-Generated List of Interlock Conditions

An interlock indication will appear in the Operator Pop-up window when an interlock condition is applied to a motor. To track the condition, the operator can simply select the interlock indication. The list is generated automatically and presented in graphical form; no configuration is necessary for this feature. To activate the condition, select the dependency icon on the operator interface pop-up window.


 
 

Pop-Up Windows

The Motor Class contains interface pop-up windows for Operation, Maintenance and Parameterization. These windows are already built into the class and no further configuration is necessary to access them. The Operator Interface pop-up window is accessed by selecting the Motor object itself. The Maintenance and Parameterization pop-up windows are accessed from the Operator Interface pop-up.

Trending

As a default setting, the Motor Class trends the output (on/off) and the run feedback. The Trend function also works in simulation mode.

User Access Configuration for Individual Instances

User access for the following functions can be uniquely configured for each instance of the Motor class:

  • Trending
  • Operations
  • Interlock
  • Service & Maintenance
  • Parameterization
See Configuration for details.

Visual Indication of Forced I/O

The Motor class continuously checks if any of its I/O variables are in "Force" mode. If so, an indication will appear on the Motor's graphical representation and the message "Forced I/O" will appear in the Popup Interface. See Implementation Notes for details.

Installation

To install the Motor class, all of the above files must be copied into a common directory within the project's "classlib" directory. The name of the directory can be chosen freely. For example (refer to the directory structure below), to add the Motor class to an existing class library follow the directory structure in the automationX program directory:

AutomationX\projects\my_project\classlib\ProcessIndustry\motor\motor_1speed

Parameterization and Mode Pop-Up

  • The Parameter section allows setup of the motor's maximum feedback response time. The default value is five seconds.
  • The Modes section allows switching to (local) Simulation Mode. In this mode all inputs are simulated and the output is disconnected from the field. The Overwrite Interlock Mode connects the auto_on variable with the field regardless of the status of the interlock variable. If neither of these two check boxes are enbled the motor is in standard mode.
  • The Emulations section allows an individual emulation of the field inputs in case they are INVALID. The Open and Closed   Input "follows" the output with a variable delay time (default = 3 seconds).
 
Maintenance Pop-Up
 

The "Enable Maintenance" Button enables or disables all maintenance functions. If one of the limits is exceeded the valve class launches a Service Request "Warning". Each Limit can be reset individually by pressing the "R" button. Upon reset, the current date is automatically entered in the according field.

Configuration

I/O Configuration

The motor is linked to the field  by filling out the reference fields of the following variables: on, run_in, rdy_in, rem_in. The other possible parameters are: the maximum time delay between a output command and the arrival of the proper feedback (error_time) and the limits for service request (maintenance pop-up).

Notes:

  • If "run_in" is not connected, the class routes the output to the internal run variable. This "fakes" that the feedback is always OK. If run_in is connected, the field must provide the proper feedback.
  • If "rdy_in" is not connected, the class assumes that the motor is always ready.
  • If "rem_in" is not connected, the class assumes that the remote switch is always turned on.
  • The maximum delay time for the run feedback can be changed (in online mode) in the parameterization pop-up window.
Naming Conventions for Automated I/O Configuration

Normally, while implementing a class containing input and output variables (VAR_INPUT, VAR_OUTPUT), the Reference Fields (of these variables) need to be configured individually. However, if a class developer is required to use specific naming conventions, he or she may know the reference names in advance. For example, an organization’s internal standards may dictate tag creation according to the following conventions:

Tag name RUN FEEDBACK = Instance Name + "_" + "RUN"
 
 


 
 

automationX allows substitution addressing. The developer enters $Path_RUN and $Path_ON in the appropriate reference fields and the system automatically configures each instance according to the production rules of the class. For example, the instance name "M1234" produces "M1234_RUN " and "M1234_ON".
 

Linking the Motor to the Process

Instances of the motor class are linked to a process via two variables: "auto_on" and "lock".

  • auto_on is the condition that indicates that the motor will run in Automatic Mode. The prerequisites are the motor is NOT in Manual Mode, and no Interlock is present.
  • lock is the condition that the motor will not run in Automatic or Manual Mode. Use the Overwrite Interlock check box in the Parameter pop-up window to overwrite an Interlock condition.
 
User Access Configuration for Individual Instances

It is possible to set up unique user permissions for individual instances of the Motor class. During configuration of an instance, locate the "access" variables "TrendAccess", "OperateAccess", "InterlockAccess", "ServiceAccess" and "ParameterAccess" in the Client Data list. Selection of these variables will open the user access settings in the attribute window. From here, choose the appropriate user permissions by enabling or disabling the check-box beside the user level.
 
 


 
 

An access variable is enabled when a user is logged in with that access level. Likewise an access variable is disabled if there is no user logged on with that access level. Access variables are attached to control elements (e.g. buttons) within the class, and are checked via ACL when deciding whether or not a user is allowed to use that control element. 

For example, the access_trend_docu variable in the previous figure shows Access Levels "Guest" and "Operator" as disabled. If a user is logged on using one of these levels and pressed the Trend button on the popup interface, access is denied according to the Button1Down callback code attached to the button.
 
 


 
 

Additionally, the access_trend_docu variable is attached to the SelectSet variable in the Client Data of the Trend button element - in this case a rectangle. When the TrendAccess variable is true, the button displays a full color trend pixmap indicating that the button is active and there is permission to use it. When the TrendAccess variable is false, the button displays a grey trend pixmap indicating that the button is inactive and permission to use it is currently denied.

Implementation Notes

Higher Class Sophistication through pre-evaluation of Inputs

The inputs RUN, REMOTE, and READY are connected to "run_in", "rem_in", and "rdy_in". However, the motor class works with internal variables "run", "rem", and "rdy". This approach allows manipulation of the variables if the simulation mode is on, or one or more input variables are not connected. The result is a much more sophisticated class.
 
 


 
 
 
 

Simulation

The Motor class has a simulation mode which can be enabled by either one of two ways:

  • Local Simulation (The class variable sim_local is ON, Activation through Parameter Pop-Up)
  • Global Simulation (The system variable "$$_simulation" variable is ON)

 
 

Alarm Supression for Invalid Field Links

The Motor class avoids the generation of unneccessary alarms in the event that some I/O loses proper connection to the field.
 
 


 
 

Checking Inputs for FORCE Mode

The input variables "run_in", "rdy_in", "rem_in" and "ack_in" are checked for "Force" mode within the Motor's FB logic by addressing the ".FORCE" attribute. This attribute is inherited from the I/O and is made available upon creation of an input variable (see the automationX System Manual for configuration details).

If any of these conditions become true a "forced_io" alarm variable is set, triggering the Forced IO indications in the visual portions of the class.
 
 


 
 

This function is built into the Motor class and requires no extra configuration to implement.
